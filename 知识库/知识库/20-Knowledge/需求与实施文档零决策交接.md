## 定义

需求、验收标准、实施总览、实施周期和最小任务卡必须面向普通模型零决策执行。强模型冻结业务、技术、测试、回滚和停止决策，执行模型只按当前任务卡行动。

## 适用范围

需求复杂度按 L1-L4 分层；L2 及以上按语义提供流程图和时序图，L3/L4 追加状态、数据、依赖或故障图。所有条件字段必须填写，或写 N/A + 原因 + 证据。

## 操作规则

- 稳定追踪链：SRC -> DEC -> REQ/RULE -> AC -> CYCLE -> TASK -> 文件/符号 -> TEST -> EVIDENCE。
- 每个 TASK 只能属于一个 CYCLE，按实现 -> 真实测试 -> 审查 -> 验收逐个闭环。
- 任务必须写精确文件/符号、测试命令、local 环境、样本、断言、清理、回滚、完成和停止条件。
- 文档交付前运行对应 profile 的 validate_engineering_docs.py；P0/P1 未决、失效链接、孤立 ID 或图文不一致阻断。

## 证据

本轮项目规则、需求/验收/实施基线、文档 profile 和正反测试均已在本地仓库落盘并通过验证。

## 2026-07-12 通用上线测试引擎计划口径

- `project-release-test-rules` 的正式执行顺序冻结为：三方契约索引对账 -> 五协议 local fixture 生命周期 -> 缺 runtime 确定性 PENDING -> P0/P1 required/result 全量执行 -> 唯一 gate -> 第二项目 baseline 复用 -> 证据互链与最终验收。
- 运行时门禁必须同时校验 local provenance、fixture lifecycle、cleanup status、strict flags、contract sync、required/result 集合和 P0/P1 状态；任何非 local、secret、cleanup failure、P0 非 PASS 或集合不等价都只能阻断。
- 计划文档必须为每个任务冻结精确文件/符号、local 命令、fixture 路径、正反断言、清理、`ROLLBACK-*`、最大推进边界和四类 EVD；validator 的 PASS 只对真实扫描范围负责。
