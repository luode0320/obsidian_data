# 微业务(伪微服务)架构

## 定义

在**一个项目、一个服务**内,把不同业务按目录包垂直切分,业务包之间互不关联(禁止横向 import);需要交互时统一经一个**公共接口包(contract)**以接口形式通信,从而在单服务内达到"伪微服务"的上下文隔离效果。

## 动机

AI 天然适合微服务式上下文——每次只读一个业务的上下文即可分析其实现逻辑。真微服务(独立进程/部署/仓库)过重;微业务在单服务内以更低成本获得同等的"单业务上下文自包含"收益。

## 核心原则

1. 业务垂直切分:每个业务是自包含目录包(`internal/business/<域>/`,内含 handler/logic/model/store)。
2. 横向零依赖:业务包 A 禁止直接 import 业务包 B 的任何内部路径。
3. 受控通信:跨业务只经公共接口包 `internal/contract/`,依赖倒置(被调用方实现并注册,调用方依赖接口)。
4. 接口按需:只为**真实跨业务调用点**建接口,无外部调用者的业务不预建接口(服从反单实现接口)。
5. 新增即新包:新业务新开目录包,旧业务包不动。
6. 单业务可读:每业务包统一 README + 全局索引,AI 只读单业务上下文即可分析。

## 目录布局(Go 优先)

- `internal/business/<域>/`:业务自包含包(handler/logic/model/store + README.md)
- `internal/business/README.md`:全局业务包索引(业务清单+职责+状态)
- `internal/contract/<域>/`:公共接口包 + `README.md`(接口契约清单:暴露方/实现方/调用方/用途)
- `internal/common/`:无状态公共支撑,可被任意业务依赖,不反向依赖业务

## 统一 md 规范(让 AI 只读单业务上下文)

- 每业务包 README:职责/边界/目录/对外接口/依赖接口/私有模型入口/关键链路
- 全局业务索引:AI 进入项目第一入口,先定位业务包再深入
- 接口契约清单:跨业务关系全局视图

## 落地机制(luode-skills 仓库)

- skill:`micro-business-architecture-rules`(新项目自动触发 + 存量守护)
- 脚本 `micro_business.py`:`init`(幂等写标记)/ `scaffold`(建业务包骨架)/ `check`(校验跨业务非法 import)
- 触发链路:新仓库首轮建议 → 用户确认 → 写标记(`CLAUDE.md` 的 `## 微业务架构约束` + `项目设计.md`)→ 据标记守护
- 触发发生在**用户新建的业务项目**里,不在 skill 存放仓库

## 相邻边界

- 分层落点(router/service/repository 等)交给 `package-structure-rules`
- 接口抽象服从 `code-readability-rules`(反单实现接口):接口仅用于真实跨业务调用点
- 目录/命名引用 `artifact-storage-rules`
